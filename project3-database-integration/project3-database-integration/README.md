## What this project does

This builds directly on Project 2. The Task Manager API now talks to a
**real MongoDB database** instead of a temporary in-memory array, using
**Mongoose** as the ORM (Object-Relational/Document Mapping layer between
the Express backend and MongoDB).

It covers everything Project 3 asks for:

- A designed database schema (`models/Task.js`)
- Full CRUD operations (Create, Read, Update, Delete) hitting the database
- Proper data handling: schema-level validation, error handling for bad
  IDs, and Mongoose's built-in protection against injection-style attacks
  (it uses parameterized queries internally, so raw string concatenation
  is never used)

## Why MongoDB + Mongoose

The slides compare SQL (tables, strict structure) vs NoSQL (flexible,
document-based). I went with **MongoDB (NoSQL)** because:
- It pairs naturally with the MERN stack I'm building toward
- A "task" doesn't need complex joins/relationships yet, so a flexible
  document model fits better than a rigid table at this stage
- Mongoose still gives me schema validation, so I get NoSQL flexibility
  without losing the safety of strict rules (required fields, min/max
  length, allowed values, etc.)

## Database Schema (Task collection)

| Field       | Type     | Rules                                              |
|-------------|----------|-----------------------------------------------------|
| title       | String   | required, trimmed, 3-100 characters                 |
| description | String   | optional, max 500 characters                        |
| completed   | Boolean  | defaults to `false`                                  |
| priority    | String   | enum: `low`, `medium`, `high` (defaults to `medium`) |
| createdAt   | Date     | auto-generated by Mongoose `timestamps`              |
| updatedAt   | Date     | auto-generated by Mongoose `timestamps`              |
| _id         | ObjectId | auto-generated primary key by MongoDB                |

This is the equivalent of the "Primary Key" idea from the slides - every
task gets a unique `_id` automatically from MongoDB.

## How to run it

1. Create a free MongoDB Atlas cluster at https://www.mongodb.com/atlas
   (or use a local MongoDB install if you have one).
2. Copy `.env.example` to `.env` and paste your connection string into
   `MONGO_URI`.
3. Install dependencies and start the server:

```bash
npm install
npm start
```

4. Open `http://localhost:5000` in your browser to use the test UI, or
   hit the API directly with Postman/curl.

> Note: I built and validated all the logic carefully, but I could not
> run a live MongoDB connection inside this sandboxed environment
> (no internet access to Atlas from here). Once you add your own
> `MONGO_URI` and run `npm start` on your machine, it will connect and
> work exactly as documented below.

## API Endpoints

| Method | Endpoint        | Description                          |
|--------|-----------------|----------------------------------------|
| POST   | /api/tasks      | Create a new task                       |
| GET    | /api/tasks      | Get all tasks (supports `?completed=true/false`) |
| GET    | /api/tasks/:id  | Get a single task by id                  |
| PUT    | /api/tasks/:id  | Update a task                            |
| DELETE | /api/tasks/:id  | Delete a task                            |
| GET    | /api/health     | Check if server is alive                  |

### Example - Create a task

Request:
```json
POST /api/tasks
{
  "title": "Submit Project 3",
  "description": "Database integration milestone",
  "priority": "high"
}
```

Response (201 Created):
```json
{
  "success": true,
  "message": "Task created successfully",
  "data": {
    "_id": "665f1c2e8b1d4a0012a3f9e1",
    "title": "Submit Project 3",
    "description": "Database integration milestone",
    "completed": false,
    "priority": "high",
    "createdAt": "2026-06-30T10:00:00.000Z",
    "updatedAt": "2026-06-30T10:00:00.000Z"
  }
}
```

Validation error (400 Bad Request) - e.g. title too short:
```json
{
  "success": false,
  "message": "Title must be at least 3 characters long"
}
```

## Notes / things I learned doing this project

- The database should never blindly trust the application - that's why
  validation lives in the schema (`models/Task.js`), not just in the
  controller. Even if I forget a check in the controller, Mongoose still
  enforces the rules.
- Mongoose automatically parameterizes queries, which is what protects
  against injection attacks (the slide's "vulnerable string concatenation
  vs secure parameterized query" comparison).
- `findByIdAndUpdate` with `runValidators: true` was important - by
  default Mongoose skips validation on updates unless you explicitly
  turn it on.
- Separating `config/db.js`, `models/`, `controllers/`, and `routes/`
  keeps the project easy to extend later (e.g. adding a `User` model and
  linking tasks to users with a foreign-key-style reference).
